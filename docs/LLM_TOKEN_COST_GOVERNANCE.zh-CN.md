# LLM Token 与成本治理

> 适用基线：Linux 8 vCPU / 16 GB RAM / 300 GB SSD 企业服务器。
>
> 边界声明：该主机只承载知识库控制面和本地数据服务，不承载 50 亿 token/日的本地模型推理。该吞吐目标仍须由独立的企业内网推理集群或经批准的模型服务提供，并提交容量实测、合同配额和成本上限证据。

## 1. 安全闭环

所有外部 LLM 调用必须按以下顺序执行：

1. 根据 `tenant / user / API key / provider / model` 匹配全部启用的硬预算策略；
2. 根据启用的模型价格计算最坏情况成本，未知价格或无预算策略时拒绝调用；
3. 在 PostgreSQL 事务内锁定策略和日/月计数器，原子预留 token 与微美元额度；
4. 提交预留后才允许请求离开企业应用边界；
5. 使用供应商返回的输入/输出 token 完成实际结算，并退回未使用预留；
6. 供应商未返回用量、返回了不同模型，或网络结果无法判断时，保留最坏情况额度等待人工对账；
7. 同一幂等键只能产生一条账本记录，并发竞争由数据库唯一约束最终仲裁。

账本只记录维度、token、价格快照、成本、状态和错误分类，不记录 prompt、知识片段、回答正文、API 密钥或请求头。

## 2. 数据与计费语义

| 对象 | 用途 |
|---|---|
| `llm_model_prices` | Provider/Model 价格目录；单位为每百万 token 的微美元 |
| `llm_budget_policies` | 租户级及可选用户、API 密钥、Provider、Model 维度的日/月硬预算 |
| `llm_budget_counters` | 在 PostgreSQL 行锁下维护的已使用与已预留聚合值 |
| `llm_usage_records` | 内容无关的预留、实际结算和异常状态账本 |
| `llm_usage_budget_holds` | 固化一次调用实际命中的策略及日/月窗口，避免策略更新改变历史结算 |

`1 USD = 1,000,000 micro-USD`。金额使用整数计算并向上取整，不使用浮点数。

## 3. 管理入口

- `PATCH /api/v1/llm/providers/{provider}`：同时配置当前模型的输入/输出价格。两项价格必须一起提交；切换到没有价格的模型后，调用会安全降级。
- `GET /api/v1/llm/budget-policies`：查看预算策略，需要 `quota:manage`。
- `POST /api/v1/llm/budget-policies`：创建租户或细分维度的日/月硬预算，需要 `quota:manage`。
- `PATCH /api/v1/llm/budget-policies/{policy_id}`：调整限额或停用策略，需要 `quota:manage`。
- `GET /api/v1/llm/usage`：查询内容无关账本，支持 Provider、Model、用户、API 密钥、知识库、状态和时间过滤，需要 `audit:read`。
- `POST /api/v1/llm/usage/{usage_id}/reconcile`：仅用于释放经人工确认已经停止外呼的长期 `held` 租约，需要 `quota:manage`。请求必须提交 `provider_egress_terminated: true` 和不少于 10 个字符的核验原因；系统会记录操作者、请求 ID、原因与账本 ID。

生产环境至少应配置一条租户总预算，并按业务风险追加用户、API 密钥和 Provider/Model 预算。多个匹配策略同时生效，任意一个策略不足都会在调用前拒绝请求。

## 4. 故障处理

| 场景 | 状态 | 额度处理 |
|---|---|---|
| 正常返回且用量完整 | `settled` | 按实际 token 结算，退回差额 |
| 上游明确拒绝且未执行推理 | `released` | 全量释放 |
| 网络超时、连接中断、未知异常 | `indeterminate` | 保留最坏情况预留，禁止同键重试 |
| 缺失用量或返回模型与价格快照不一致 | `indeterminate` | 保留预留并进入人工对账 |
| 幂等键重复 | 原记录状态不变 | 不再次调用供应商 |

`indeterminate` 不得通过直接删除账本或修改计数器处理。当前 reconcile 入口只接受 `held` 状态，用于进程崩溃后遗留的外呼租约；操作者必须先通过供应商控制台、内部推理网关或网络连接记录确认请求已经终止。`indeterminate` 仍须与供应商账单或内部推理网关请求记录核对后，走独立的财务对账流程，不得使用该入口释放。

## 5. 外呼租约与撤权顺序

聊天生成、独立答案审核和 OKF 转换在外呼前先提交 `held` 账本，再按知识库、用户、API Key 和当前角色维度取得排序后的 PostgreSQL 事务级 advisory lock，并重新解析实时权限、知识库访问级别和外部处理同意。用户停用、用户角色替换、角色权限/策略替换、知识库角色授权替换、知识库外呼同意关闭和 API Key 撤销使用同一命名空间的锁，再检查相同的活动租约：

- 没有活动租约时，撤权按正常流程提交；
- 有活动租约时，撤权返回 HTTP `409` 和固定错误码 `external_llm_processing_in_progress`，并记录拒绝审计；
- 外呼结算后，管理员或密钥所有者重试撤权；
- 不允许通过直接修改 `llm_usage_records`、删除预算 hold 或修改计数器绕过冲突。

锁键由版本化命名空间、资源类型和 UUID 计算为稳定的有符号 64 位值；多维锁按数值排序以避免死锁。哈希碰撞只会让无关请求安全地串行，不会绕过授权。事务在最终授权检查完成后立即提交并释放锁，锁不会跨越模型网络等待。

进程异常退出导致 `held` 长期存在时，先确认供应商侧请求已结束，再由具备 `quota:manage` 的人员调用 reconcile。该操作属于高风险运维动作，应由第二人复核审计日志。

## 6. 验收边界

当前自动化证据覆盖：SQLite 行为回归、PostgreSQL 并发幂等与预算防双花、真实 PostgreSQL 迁移升级/降级、管理权限、OpenAPI、RAG 生成与语义审核双调用、OKF 转换调用，以及 Ruff/MyPy。

这些证据证明控制面具备最小安全闭环，但不证明 50 亿 token/日能力。最终 `CAPACITY-P0-001` 仍必须保持阻断，直到目标推理集群完成平均约 57,870 token/s、业务窗口约 173,611 token/s 以及约 868,056 token/s 五倍峰值的真实容量验收。
